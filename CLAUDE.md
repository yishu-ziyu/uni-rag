# uni-rag 项目开发守则

## 默认行为：所有开发从 Ship 出发

收到任何改动请求（包括"改一下"），先过 ship-align，ship 自己判断需要走几步。

## Ship 决策门（改动前必答 4 题）

1. 谁在使用？他在做什么任务？
2. 改什么？明确不改什么？
3. 怎么知道做对了？（验收标准）
4. 需要几只手并行？

**≤ 30 秒能回答前两题** → 小改动，直接做
**第三题说不清楚** → 先写验收标准，再动手
**第四题 > 1** → 进 /loop

## 跳过规则（以下情况不走 ship）

- Typo、1 行 bug 修复、配置文件修正
- 改完能立刻验证，且不需要记录"为什么"

## 必须走 Ship 的情况

- 涉及 UI 改动
- 涉及 2+ 文件
- 新功能或新组件
- 改动原因需要下周的你还记得

## 主题：DataFlow 对 UniRag 的适配性调研

### 背景
用户在做一个 RAG 产品 UniRag（~/Desktop/产品项目学习/uni-rag/），v0.1 已交付（96% 测试覆盖率）。调研 DataFlow（北大 OpenDCAI 的数据准备框架）对 UniRag 的价值。

### 当前数据流（已读源码确认）

```
文件上传（PDF/DOCX/MD）或 URL
    ↓
提取层
  - PDF: PyMuPDF → text + pages[]
  - DOCX: python-docx + 公式包装
  - MD: MarkdownIt → 去 HTML
  - URL: Trafilatura + httpx → 正文提取
    ↓
分块层（chunker.py）
  - 按 # header 切 section
  - 每 section 按 1000 字符切
  - 输出 Chunk(text, section_title, start_offset, end_offset, page_number)
    ↓
嵌入层（embedder.py）
  - BGE-M3, 1024 dim, multilingual
    ↓
索引层
  - VectorStore (Chroma)
  - BM25Index
```

### DataFlow 的核心能力（已读 GitHub README 确认）

1. **算子化设计** — Pipeline → Operator → Prompt 分层，100+ 预置算子
2. **Knowledge Base Cleaning Pipeline** — 从原始文本提取结构化条目，清洗 RAG 知识库
3. **QA 合成流水线** — 从文档生成高质量 QA 对（用于 SFT / RL / 评估）
4. **评估 + 过滤算子** — step-by-step 验证、多模型交叉打分
5. **精炼算子** — 提升问题难度和考察深度
6. **依赖 MinerU 做 PDF 提取**（不是自己做）

### 关键发现

**DataFlow 解决的是"有内容的文本 → 高质量训练/评估数据"的问题。**
UniRag 解决的是"文档 → 可检索的知识库"的问题。

两者的交集在 **数据入库质量** 这一层。

### UniRag 当前缺失的能力

| 缺失能力 | 当前表现 | 影响 |
|----------|----------|------|
| 提取质量过滤 | Trafilatura 提取后直接 chunk，无质量检查 | 噪音/短文本/无关内容入库 |
| Chunk 质量评估 | 所有 chunk 一视同仁嵌入 | 低质量 chunk 污染检索结果 |
| RAG 评估集 | 只有功能测试（BDD），没有检索质量测试 | 不知道检索好不好 |

### DataFlow 可直接借鉴的思路

1. **提取后质量门控** — 在 chunking 之前加一个轻量过滤（太短？纯噪音？重复？）
2. **Chunk 质量评分** — 用 LLM 给每个 chunk 打个分，低于阈值的降权或丢弃
3. **评估集生成** — 从已有文档库生成 QA 对，定期跑 retrieval eval

### 引用系统详细分析（locator.py + pipeline.py）

当前引用验证流程：
1. LLM 输出里用 regex `\[([a-zA-Z0-9_]+:\d+)\]` 提取引用标记
2. 在 chunk_map 里找到对应 chunk
3. 用 `locate_citation()` 在原始文件里做模糊匹配找 span

问题：
- 只验证了"LLM 引用的 chunk_id 存在"，不验证"LLM 声称的内容真的在这个 chunk 里"
- 没有回检机制：如果 LLM 编造了一段话然后挂了一个真实 chunk_id，系统不会发现
- `locate_citation` 用前 30 字符做模糊匹配，精度有限

### 回答风格分析（prompts.py）

SYSTEM_PROMPT 是硬编码的学术腔：
- 固定称呼"学术助手"
- 要求每条事实都带引用
- 强制中文回答

对所有场景一视同仁，没有适配机制。

## Looper 评测闭环设计（2026-06-24 晚）

在并行完成三路质量改进（数据过滤 / 风格适配 / 引用验证）后，我们配置了 Looper 作为自动化的质量裁判系统。

### 1. 目标 (Goal)
提供一个可重复运行的验收环境，证明三路改进对最终 RAG 回答确实有效，若退化则自动尝试修复。

### 2. 评测框架配置 (loop.yaml)
- **触发命令**: 针对特定测试词("解释一下什么是量子计算")发起带风格参数的真实 query。
- **评委选型 (Council)**: 使用单模型 `claude-sonnet-4-6` 担任 LLM-as-judge，平衡成本与准确度。
- **打分机制 (Verification)**:
  - 准确性 40%
  - 引用格式与准确性 35%
  - 风格匹配度(academic) 25%
  - *阈值*: 总分 ≥7.5 且单项不低于 7.0 判定为 PASS。
- **自动迭代 (Control)**: 
  - `max_iterations: 3` (防止无限调优浪费 token)
  - 严格限制 `tunable_files` 仅为 `prompts.py` 和 `config.py`，确保核心逻辑安全。

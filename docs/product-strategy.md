# uni-rag 产品策略

> 2026-06-22 · 学生 RAG 学习工具的长期产品方向

---

## 一、产品愿景（North Star）

uni-rag 不只是一个"能对话的 PDF 阅读器"，而是**学生个人的第二大脑工作台**——把散落在课程 PDF、讲义、网页、笔记里的知识，自动编译成可检索、可关联、可追问的知识网络。

核心愿景可以用一句话概括：

> **Compile once, query many.** 上传一次材料，终身带着原文依据提问。

终极形态是 Karpathy LLM Wiki 的三层架构在本地运行：
- **Raw Sources**：原始文件（PDF、网页、DOCX、Markdown）完整保留
- **Wiki Layer**：LLM 自动生成的概念页，用双链互相引用
- **Schema Layer**：学生自己提炼的规则、框架、公式，作为顶层索引

对学生而言，这意味着期末复习时不用重新翻整本教材——问"这学期微观经济学里边际成本讲了什么"，答案直接带章节引用和关联概念。

## 二、差异化定位

| 竞品 | uni-rag 的差异 |
|------|---------------|
| **Side sider** | Side sider 是通用知识库工具，定位模糊。uni-rag 只服务**学生场景**——课程材料、考试复习、论文写作，所有功能围绕这个场景优化 |
| **Obsidian** | Obsidian 是笔记工具，知识库需要用户手动建立。uni-rag 是**自动化的**——上传即索引、自动切块、自动生成概念页，零手动维护成本 |
| **ChatPDF** | ChatPDF 是单文件对话玩具。uni-rag 支持**多文件跨文档问答**、知识库隔离、URL 提取，而且**本地运行**，不依赖云端 |
| **Humata** | Humata 付费墙高，且文档留在其服务器。uni-rag 完全**本地优先**，数据不出本机，对学生隐私敏感群体（论文、未公开材料）是刚需 |
| **Dify / FastGPT** | 这些是面向开发者的 RAG 平台，配置复杂。uni-rag 开箱即用，**零配置**（除了 API key），CLI + Web 双入口 |

**一句话定位**：本地运行、开箱即用的学生知识库 RAG 工具。

## 三、核心功能优先级

### P0 — 必须有的初始吸引力

1. **多格式上传 + 自动索引**
   - PDF、Markdown、DOCX、URL（已完成）
   - 上传后自动 chunk + 向量化 + BM25 混合索引
   - 进度条展示解析/切块/向量化/写入四阶段

2. **带引用的问答**
   - 多轮对话，上下文保持
   - 每个回答附可点击引用，点击后跳转原文高亮位置
   - 引用 chip 显示文档名 + 页码/段落位置

3. **上传后建议问题**
   - 上传完成后，LLM 自动生成 3 个引导问题
   - 用户点击直接追问，降低首问门槛
   - 实现成本极低，但显著提升首次体验

4. **知识库隔离**
   - 支持多知识库（按课程/主题分组）
   - 切换知识库清空对话但保留文件

5. **导出答案**
   - PDF / Markdown 导出（已完成）

### P1 — 差异化亮点

1. **自动 Summary 卡片**
   - 上传完成后，LLM 生成 2-3 句摘要
   - 显示在文件列表旁，用户不点进去就能了解内容概况
   - 对 50+ 页的教材特别有用

2. **追问建议流**
   - 每次回答后，LLM 额外生成 2-3 个追问建议
   - 形成线程式研究流，类似 Perplexity 的推荐追问

3. **引用高亮精度优化**
   - 当前引用是近似匹配，需要提升到精确段落定位
   - 在 doc panel 中高亮显示被引用的具体段落

4. **本地知识图谱（轻量版）**
   - 基于 LLM 提取的概念关系，不追求完美图谱
   - 展示"这篇文章提到了哪些概念"、"这些概念在哪些文件里出现过"
   - 图谱节点 = 概念，边 = 共现关系，点击跳转到包含该概念的回答

### P2 — 增长功能

1. **LLM Wiki Layer（核心差异化）**
   - 上传一组材料后，自动编译成 Wiki 页面
   - 每个概念一个页面，包含定义、来源引用、相关概念双链
   - "Compile once, query many"——查询时优先返回 Wiki 页面而非原始 chunks

2. **Lint / 自愈机制**
   - 定期扫描知识库，发现孤立概念页、矛盾定义、过期引用
   - 自动修复或标记待人工确认

3. **知识图谱可视化**
   - 三栏布局的 Graph View（参考 Obsidian）
   - 节点大小 = 引用密度，颜色 = 概念类型
   - 可拖拽、缩放、点击聚焦

4. **浏览器插件 / 快捷输入**
   - 浏览器侧边栏，选中网页文字直接发送到 uni-rag
   - 降低从网页到知识库的录入成本

5. **多用户 / 共享知识库**
   - 班级/学习小组共享知识库
   - 权限控制：谁能上传、谁能提问

## 四、架构原则

### 核心原则：后端 Lego 化，前端不关心存储

```
┌─────────────────────────────────────────┐
│              Web UI / CLI                │
│         (薄前端，只发 HTTP 请求)          │
├─────────────────────────────────────────┤
│           FastAPI 路由层                 │
│     (ingest / query / kb / export)       │
├──────────┬──────────┬──────────┬─────────┤
│ Ingest   │ Retrieve │  LLM    │  Export │
│ Pipeline │ Pipeline │ Client  │  Module │
├──────────┴──────────┴──────────┴─────────┤
│          抽象接口层                        │
│  (Chunker / Parser / Embedder / Vector) │
├──────────┬──────────┬──────────┬─────────┤
│ ChromaDB │ BM25     │ Trafil. │ Weasy.  │
│ 本地向量 │ 本地全文  │ URL提取  │ PDF导出  │
└──────────┴──────────┴──────────┴─────────┘
```

**三条铁律**：

1. **接口优先**：每个模块对外暴露抽象接口（ABC），内部实现可替换。例如 `Embedder` 接口，可以从 sentence-transformers 换到本地 Ollama，前端无感知。
2. **单一数据源**：所有状态存在 SQLite（`kbs.db` + `sessions.db`），不依赖外部服务。Docker 卷持久化，删除容器不丢数据。
3. **无状态 API**：每个 HTTP 请求自包含，session 通过 cookie 或 header 传递，方便水平扩展。

### 当前架构已满足 Lego 化的部分

- `ingest/pipeline.py`：解析 → 切块 → 向量化 → 写入，流水线化
- `rag/pipeline.py`：检索 → 重排 → LLM 生成，同样是 pipeline
- `store/`：向量库、BM25、知识库管理，已抽象

### 需要继续 Lego 化的部分

- **Parser 层**：目前 PDF/DOCX/MD/URL 各一个 parser，需要统一成 `DocumentParser` 接口
- **Extractor 层**：LinkExtractor 注册表模式已经做了很好的抽象，可以扩展到更多平台
- **LLM Client**：当前硬编码 MiniMax/Anthropic 兼容协议，需要支持多 provider（OpenAI、本地 Ollama、DeepSeek）

**三栏逻辑**：
- **左栏（文件树）**：知识库内的文件列表 + URL 快捷添加 + 知识图谱缩略入口
- **中栏（对话区）**：聊天主区域，下方固定输入框

  1. 原文阅读（点击引用后显示高亮段落）
  2. 知识图谱（轻量可视化）
  3. 知识库管理（KB 列表 + 设置）

- 点击引用 chip → 右栏切到原文模式，自动滚动到高亮位置
- 点击追问建议 → 直接作为新消息发送
- 点击文件树中的文件 → 右栏显示该文件的 Summary 卡片
- 右键文件 → 重命名 / 删除 / 查看关联概念

## 六、知识图谱实现方案

### 不做完美的 Neo4j，做够用的本地图谱

uni-rag 的知识图谱不需要像学术图谱那样复杂。学生场景下，**概念共现关系**就够了。

### 数据模型

```
ConceptNode {
  id: str           # 概念唯一标识（LLM 生成的规范化名称）
  name: str         # 显示名称
  type: str         # 概念类型：理论/人物/公式/方法/术语
  source_files: list # 出现在哪些文件中
  citation_count: int # 被引用次数（决定节点大小）
  wiki_page_id: str  # 关联的 Wiki 页面（如果有）
}

ConceptEdge {
  source: str       # 源概念 ID
  target: str       # 目标概念 ID
  relation: str     # 关系类型：属于/对比/因果/依赖/示例
  strength: float   # 共现强度（0-1）
}
```

### 提取方式

1. **上传时**：LLM 从每个 chunk 中提取提到的概念，建立 chunk → concept 的映射
2. **定期批处理**：对知识库内所有 chunk，LLM 提取概念关系，更新图谱
3. **Wiki 生成时**：生成概念页时顺便提取与其他概念的关系

### 图谱展示

- **轻量模式**（P1）：在对话区侧边显示"相关概念"小卡片，不追求全貌
- **完整模式**（P2）：右栏展开为可交互图谱，用 D3.js 或 Cytoscape.js 渲染
  - 节点大小 = citation_count
  - 节点颜色 = type（理论=蓝色，人物=紫色，公式=橙色...）
  - 边的粗细 = strength
  - 拖拽、缩放、点击聚焦
  - 点击节点 → 右栏切换到该概念的 Wiki 页面

### 存储

概念和关系存储在 SQLite 的 `concepts` 和 `concept_edges` 表中，不单独建图数据库。图谱查询用 SQL JOIN + 内存渲染，对 < 1000 个概念的性能完全够用。

## 七、LLM Wiki Layer

### 三层架构

```
┌─────────────────────────────────────────────┐
│  Schema Layer（规则层）                       │
│  "微观经济学：边际成本递增规律"               │
│  ← 链接 → "微观经济学：机会成本"              │
│  ← 链接 → "经济学十大原理"                    │
├─────────────────────────────────────────────┤
│  Wiki Layer（概念页层）                       │
│  每个概念一页，LLM 编译自 Raw Sources          │
│  - 定义                                    │
│  - 来源引用（[教材.pdf p.45]）               │
│  - 相关概念双链（[[机会成本]]）               │
│  - 学生笔记（可手动编辑）                     │
├─────────────────────────────────────────────┤
│  Raw Sources（原始资料层）                     │
│  原始文件全文 + chunk 向量化存储               │
└─────────────────────────────────────────────┘
```

### 编译流程（Ingest → Compile）

1. **用户上传一组材料到知识库**（比如 CS101 整学期 PDF）
2. **后台自动编译**：
   - LLM 读取所有 chunk，提取核心概念（目标 20-50 个）
   - 对每个概念生成 Wiki 页面（定义 + 来源 + 双链）
   - 存储在 `wiki_pages` 表中，格式为 Markdown
3. **查询时优先返回 Wiki 页面**：
   - 用户问"边际成本是什么？"
   - 先在 Wiki 层查找"边际成本"页面
   - 找到 → 直接返回页面内容 + 关联概念
   - 没找到 → 降级到 Raw Sources 检索，同时触发"该概念值得建页"的提示

### Wiki 页面格式

```markdown
# 边际成本

**定义**：生产每增加一单位产品所增加的总成本。

**来源**：
- [教材.pdf p.45-47]
- [讲义3.md §2.1]

**相关概念**：
- [[机会成本]]：决策时考虑的成本类型
- [[规模经济]]：边际成本递减的条件
- [[供给曲线]]：边际成本曲线的应用

**学生笔记**：
（可编辑区域，支持手写或打字）
```

### 双链实现

- Wiki 页面内用 `[[概念名]]` 语法
- 前端渲染时自动转为可点击链接
- 反向链接面板：查看"哪些页面链接到当前页面"（参考 Obsidian 的 Reverse Links）

## 八、Lint / 自愈机制

### 为什么需要 Lint

Wiki 层最大的风险是**内容腐烂**：
- 概念定义过时（教材版本更新）
- 双链断裂（概念被重命名后旧链接失效）
- 孤立页面（建立了概念页但没有来源引用）
- 矛盾定义（同一概念在不同 chunk 中有冲突描述）

### Lint 触发时机

1. **定时扫描**：每天凌晨对知识库跑一次 Lint
2. **事件触发**：上传新文件、删除文件、手动重建 Wiki 后立即跑 Lint
3. **用户主动触发**：设置面板里"检查知识库健康度"按钮

### Lint 规则清单

| 规则 | 检测逻辑 | 修复策略 |
|------|---------|---------|
| **孤立 Wiki 页** | Wiki 页面无任何来源引用 | 自动删除，或标记待确认 |
| **断裂双链** | `[[概念名]]` 在知识库中不存在 | 自动修正为最近匹配，或标记为待处理 |
| **矛盾定义** | 同一概念在不同来源中有冲突描述 | 列出冲突，LLM 生成综合定义，用户确认后更新 |
| **过期引用** | 来源文件已被删除，但 Wiki 仍引用它 | 标记引用为失效，建议重新编译 |
| **低质量页面** | 概念页内容 < 50 字，或定义模糊 | 标记为待完善 |
| **未使用概念** | 概念存在于图谱中，但从未被问答命中 | 纳入"可清理"候选列表 |

### 自愈流程

```
Lint 扫描 → 发现问题 → 分类处理

自动修复（无需用户确认）：
  - 断裂双链 → 模糊匹配最近概念，修正链接
  - 孤立页面 → 如果 < 30 天且无引用，自动删除

标记待处理（通知用户）：
  - 矛盾定义 → 在知识库管理页显示冲突列表
  - 过期引用 → 建议"重新编译 Wiki"
  - 低质量页面 → 建议"完善概念页"

上报（需要 LLM 判断）：
  - 概念重命名建议（"机会成本"和"机会代价"可能是同一概念）
  - 概念合并建议（两个高度相关的概念可以合并）
```

### 动画

```
过渡：
- 全局 transition: 0.2s cubic-bezier(0.4, 0, 0.2, 1)
- 面板展开/收起：0.3s ease-out
- 按钮 hover：0.15s ease

微交互：
- 引用 chip hover → 轻微上浮 + 边框高亮
- 文件项 hover → 背景色渐变

- 不在文字上做动画
- 不在滚动容器上做视差
```

## 附录：设计决策记录

### 为什么不做云端同步？

学生场景下，数据隐私是核心诉求（论文草稿、未公开课程材料）。本地优先不是技术限制，是产品定位。云端同步可以作为 Phase 3 的可选功能，但默认必须是本地。

### 为什么 Wiki 层用 Markdown 而非数据库？

Markdown 的最大优势是**可移植性**。用户可以用任何编辑器打开、修改、导出。即使 uni-rag 停止维护，知识库内容仍然可用。这是 Karpathy Wiki 方案的核心智慧。

### 为什么知识图谱不用图数据库？

Neo4j 太重了。学生知识库 < 1000 个概念时，SQLite + 内存渲染完全够用。等概念数超过 5000 再考虑升级——大概率不会发生。

### 为什么前端不用 React/Vue？

当前 uni-rag 前端是 vanilla JS + 原生 CSS，对项目规模（< 2000 行）完全够用。引入框架会带来构建复杂度、学习成本、依赖管理，对 solo project 是过度工程。等组件数超过 20 个再考虑迁移。
